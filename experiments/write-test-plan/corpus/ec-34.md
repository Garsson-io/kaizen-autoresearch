A branch accumulated several committed changes but never got a PR. The workflow had no automated signal for "work is done but unPRed", so completed work silently went stale. The team wants detection in both session bootstrap and status reporting.

## Behaviors to cover

1. Branch staleness detection correctly identifies branches with commits not in main, older than the configured age threshold, and no open PR.
2. Session-start setup emits a stale-work warning with actionable branch context when the current branch is stale.
3. A local status-classifier function maps computed branch state to labels (`clean`, `stale-no-pr`) deterministically.
4. `/kaizen-wip` against live repo and PR state distinguishes "clean" from "has commits but no PR (STALE)" in rendered output.
5. Branches with open or merged PRs are not flagged as stale-no-PR false positives.
6. Staleness detection remains correct across rebases or squash merges where commit hashes change but branch state should no longer be classified as unPRed stale work.
