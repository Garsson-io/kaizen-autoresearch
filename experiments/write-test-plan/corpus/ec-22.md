A multi-tier caching layer sits between the application and a PostgreSQL database. It generates cache keys from query parameters, computes TTL values based on data freshness rules, selects an eviction strategy (LRU or LFU) depending on memory pressure, and invalidates entries when the underlying data changes.

## Behaviors to cover

1. Cache keys generated from identical query parameters are always identical, and keys from different parameters never collide.
2. The TTL for a cache entry is computed correctly based on the data category's configured freshness rule (e.g., user profiles = 5 minutes, product catalog = 1 hour).
3. When memory usage exceeds the configured threshold, the eviction strategy switches from LRU to LFU as specified in the configuration.
4. A write to the underlying PostgreSQL table invalidates all cache entries whose keys match the affected rows.
5. The eviction policy removes the correct entries — least-recently-used under LRU, least-frequently-used under LFU — when the cache is full.
