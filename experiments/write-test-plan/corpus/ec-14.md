A meeting transcription service receives an audio file, sends it to a speech-to-text API, then passes the transcript to an LLM to produce a structured summary with action items, key decisions, and attendee attributions.

## Behaviors to cover

1. The speech-to-text output preserves speaker labels so that the summary can attribute statements to specific attendees.
2. Transcript parsing extracts explicit task assignments (assignee + task text) into the summary input structure.
3. The live summary output contains at least one action item for every attendee who was assigned a task during the meeting.
4. Decision extraction logic routes identified decisions into the `decisions` section schema rather than generic narrative fields.
5. Key decisions mentioned in the transcript appear in the live summary's `decisions` section, not buried in the narrative.
6. When the audio file is longer than 60 minutes, the service splits it into segments and processes them sequentially, merging the results into a single coherent summary.
7. The full pipeline (upload audio, transcribe, summarize, deliver structured output) completes without manual intervention for a real meeting recording.
