A patient intake system for a clinic collects patient information, validates insurance eligibility, performs AI-assisted symptom triage to suggest urgency level, schedules an appointment, and sends a confirmation. The insurance check calls an external payer API. The symptom triage asks an AI medical reasoning engine to assess urgency based on reported symptoms. Scheduling writes to a shared calendar database.

## Behaviors to cover

1. Required patient fields (name, date of birth, insurance ID) are validated before any downstream processing and missing fields produce a clear error listing all missing items.
2. Date of birth validation rejects future dates, dates more than 150 years ago, and non-date strings.
3. The insurance eligibility check sends the correct patient ID and plan code to the payer API and correctly interprets `eligible` vs `not eligible` vs `pending`.
4. When the payer API is unreachable, the system marks eligibility as `pending` and continues the intake rather than blocking.
5. Triage invocation includes the reported symptoms and configured emergency rubric in the model input payload.
6. The live AI symptom triage assigns high urgency to symptom combinations that indicate emergency conditions (for example chest pain plus shortness of breath).
7. The triage result parser stores a clinician-readable explanation field (not only a numeric score).
8. The live AI symptom triage explanation cites case-specific symptoms and urgency rationale, not generic boilerplate.
9. An appointment is scheduled in the first available slot that matches the assigned urgency level: high urgency gets same-day, medium gets within 48 hours.
10. The scheduling step does not double-book a time slot even when two patients are processed concurrently.
11. The confirmation message includes patient name, appointment time, provider name, and preparation instructions appropriate to visit type.
12. Pipeline orchestration (validate, insurance, triage, schedule, confirm) completes correctly with deterministic stubbed triage output.
13. For a single real patient submission using only the data provided in that submission, the full live intake pipeline (validate, insurance, triage, schedule, confirm) completes without manual intervention.
14. For a returning patient case, the full live intake pipeline uses wider context (for example prior chart/history and cross-step state) so downstream decisions remain consistent with earlier triage and scheduling outputs, without manual intervention.
