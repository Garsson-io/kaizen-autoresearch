A hook I/O investigation is testing whether markdown-heavy payloads (code fences, backticks, `$vars`) can cause JSON parsing failures in hook input. Existing tests used intentionally malformed payloads with trailing junk, so the real failure mode is still unclear.

## Behaviors to cover

1. Valid JSON payloads containing markdown-heavy strings parse successfully and proceed through normal hook input handling.
2. Truly malformed JSON payloads produce a `json_parse_failed` trace record with enough context to diagnose the malformed section.
3. The local error-policy path maps parse failure to a handled result (no throw) and marks telemetry intent.
4. In real process execution, malformed stdin does not crash the hook process and still emits the expected trace signal before exit.
5. A real hook invocation path (stdin -> parser -> trace sink) produces the expected trace artifact when malformed input is supplied.
6. Regression tests distinguish malformed-JSON failures from valid-markdown payloads, preventing false attribution to markdown content itself.
