A TypeScript library wraps a third-party REST API that enforces per-account rate limits. Multiple callers within the same process share the same account quota. The library must handle quota exhaustion gracefully and expose a simple async `call()` method.

## Behaviors to cover

1. The wait time between retries doubles with each consecutive failure, up to a configured maximum.
2. The retry logic correctly identifies a rate-limit response and computes the appropriate wait time before the next attempt.
3. After the wait, the client actually resends the original request and returns the successful result to the caller transparently.
4. After the configured maximum number of retries, the client returns a typed `QuotaExhaustedError` rather than a generic error.
5. The full retry chain — initial call, backoff retries, quota tracking — terminates correctly and does not loop indefinitely before surfacing the error.
6. Requests issued concurrently from multiple async callers in the same process all draw from the same shared quota bucket.
7. When multiple requests are waiting to be sent, they are dispatched in the order they were originally submitted.
